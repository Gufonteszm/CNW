1. O Conceito de Rede e DNS
Para os containers se comunicarem, o Docker cria uma rede virtual. Em vez de você configurar endereços IP (que são voláteis), você usa o nome do serviço. Se no seu arquivo de configuração o banco de dados se chama db, sua aplicação o encontrará apenas digitando db na string de conexão. O Docker resolve o nome para o IP correto automaticamente.

2. Isolados vs. Orquestrados
Rodar containers isolados com docker run é um processo manual e desconectado; você precisa criar a rede e conectar cada container individualmente toda vez.

Já a orquestração (como o Docker Compose) automatiza tudo. Você define o sistema inteiro em um arquivo YAML e, com um único comando, ele cria a rede, sobe os serviços na ordem certa e garante que todos se enxerguem. É a diferença entre montar um rádio peça por peça ou apenas ligar a chave geral.

3. Exemplo de Fluxo no Docker Compose
No arquivo docker-compose.yml, você define serviços como app e database.

O Compose coloca ambos na mesma rede padrão.

O serviço app pode ter uma variável de ambiente DB_HOST=database.

Você pode usar a instrução depends_on para avisar ao Docker que a app só deve iniciar após o database estar rodando.

4. Problemas Comuns e Como Evitar
O erro do "Localhost": Dentro de um container, localhost refere-se a ele mesmo, não ao seu computador ou ao outro container. Para falar com outro serviço, use sempre o nome do serviço.

Corrida de Inicialização: Às vezes o container do banco sobe, mas o banco de dados ainda está processando o início. A aplicação tenta conectar e falha. A solução é configurar a aplicação para tentar novamente (retry) até que a conexão seja estabelecida.

Portas Externas vs. Internas: Dois containers podem usar a porta 80 internamente sem conflito, pois estão em IPs diferentes na rede virtual. O conflito só ocorre se você tentar mapear ambos para a mesma porta do seu computador físico.
